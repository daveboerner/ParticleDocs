    Introduction

    Particle Health supports a RESTful Web Service based off the FHIR and CCDA standards. We expose an API by which verified customers (data seekers) may access health records for over 250M unique patients across the U.S.

    We have language bindings in Shell, JavaScript and Python! You can view code examples in the dark area to the right, just select the tab of your prefered language.

    Authentication

    To authorize, use this code:

    import requests

url = 'api_endpoint_here'
headers = {'Authorization': 'token'}
r = requests.get(url, headers=headers)

    # With shell, you can just pass the correct header with each request
curl "api_endpoint_here" -H "Authorization: token"

    // Be sure to install node-fetch : $ npm install node-fetch
var url ='api_endpoint_here';
var headers = {
  "Content-Type": "application/json",
  "Authorization": "token"
}

    Make sure to replace token with your API key.

    When making requests to the Particle API, the data seeker must include an authorization token in the HTTP Header of the request like:

    Authorization: token

    You must replace token with your personal API key.

    Only accredited data seekers will be credentialed to perform queries against the Particle network.

    POST Query

    A query operation with a body containing the purpose of use and NPI. The query operation is run after it is posted and the Person ID will be returned along with a URL for checking the status of the query. Only Person resources may be queried.

    import requests

url = 'https://sandbox-dot-particle-health-staging.uc.r.appspot.com/R4/Person/{person_id}/$query'
headers = {"Content-Type": "application/json",
'Authorization': 'token'}
data = {
  "resourceType": "Parameters",
  "parameter": [{
    "name": "purpose",
    "valueString": "TREATMENT"
  }]
}
r = requests.post(url, headers=headers, json=data)

print(r.json())

    GET Query

    A query operation and its status. 202 Accepted is returned for queries that are still in progress along with the status of the query in the X-Progress HTTP header. A list of the FHIR resources found so far will also be returned in the body for queries in progress. 200 Success is returned for queries that have successfully completed along with the complete list of found resources in the body. 500 Internal Server Error is returned for queries which have failed and includes details about the failure in the body. Only Person resources may have their query status checked.

    import requests

url = 'https://sandbox-dot-particle-health-staging.uc.r.appspot.com/R4/Person/{person_id}/$query'

    GET Resources

    A search for a resource type and the specified search params. Both type and params are optional. Specifying just a type will return all resources of that type, specifying just params will perform a search across all resource types, and specifying neither will return all resources. Certain search params are supported by all resources (FHIR Search), while others are specific to resource types and are documented in the FHIR documentation for that resource type (e.g. Encounter Search).

    Architecture

    By simply requesting information with a minimum set of demographic information, Particle is able to query partner Health Information Networks (data holders), producing aggregated data in a seamless, efficient and HIPAA compliant manner.

    The demographics to include in the initial POST operation body include:

    • Last Name - family_name
    • First Name - given_name
    • Date of Birth - date_of_birth
    • Zip - postal_code - postal_code
    • Gender - gender
    • Purpose Of Use - purpose_of_use
    • NPI : National Provider Identifier - npi (optional - required for treatment based queries)
    • City - address_city
    • State - address_state
    • Address - address_lines (optional - success rate increase of ~30%)
    • Phone - telephone (optional)
    • Social Security Number - ssn (optional)

    Particle has designed this process with security, simplicity and elegance as central tenets. What took numerous coordinated IHE and RESTful queries across numerous networks has been distilled to a simple API to access data across the health ecosystem - regardless of geographic boundaries or vendor systems.

    Errors

    The Particle API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The query/file requested is hidden for administrators only.
    404 Not Found -- The specified query/file could not be found.
    405 Method Not Allowed -- You tried to access an invalid method.
    406 Not Acceptable -- You requested a format that isn't json.
    410 Gone -- The file requested has been removed from our servers.
    412 Precondition failed -- The zip file isn't ready yet.
    429 Too Many Requests -- You've made too many requests too fast.
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
